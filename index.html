Week 1 – Arrays, Sorting & Big-O
Refresh arrays, loops, and how to think about time complexity.
>

This week is about getting comfortable again with arrays, basic sorting, and describing how slow or fast an algorithm is using Big-O. You’ll solve a few simple problems and let AI critique and improve your code.

Goals:
  • Refresh arrays, lists, and basic for-loops.
  • Review bubble, insertion, and selection sort.
  • Describe runtime using Big-O notation.
Practice:
  • Solve 3 array problems: find min/max, reverse array, linear search.
  • Ask AI: “Explain the time complexity of my solution and suggest a cleaner version.”
  • Rewrite one solution after AI feedback focusing on clarity and naming.
Learn more:
  • Khan Academy – Big-O notation basics
  • Khan Academy – Algorithms track (sorting & complexity)
Answer keys / examples:
  • GitHub – “Arrays 101” solved LeetCode problems
  • Blog – Arrays 101 solutions & explanations
Quick quiz
  • Q1. What does Big-O measure: actual runtime in seconds or how it grows with input size?
  • Q2. What is the time complexity of linear search on an array of n elements?
  • Q3. Name one difference between bubble sort and insertion sort.
Show quiz answers
  1. It measures how the runtime grows with input size, not raw seconds.
  2. O(n).
  3. Bubble sort repeatedly swaps adjacent out-of-order items; insertion sort builds a sorted prefix by inserting each new element in the right place.
Week 2 – Recursion Basics & Problem Decomposition
Learn how functions can call themselves and how to think in smaller sub-problems.
>

Recursion is a core thinking tool. You’ll practice defining base cases, the recursive step, and tracing the call stack. AI will help you spot infinite loops and suggest iterative versions.

Goals:
  • Understand base case vs recursive case.
  • Implement classic recursive functions (factorial, Fibonacci, sum of array).
  • Use AI to check if your recursion is safe and efficient.
Practice:
  • Write recursive versions of factorial, Fibonacci, and sum of an array.
  • Then ask AI: “Rewrite this iteratively and explain which version is more efficient.”
  • Draw the recursion tree for factorial(4) or Fibonacci(5).
Learn more:
  • GeeksforGeeks – Introduction to Recursion
  • GeeksforGeeks – Recursive Algorithms
Answer keys / examples:
  • GitHub – Recursion problems with detailed solutions
  • Article – 20 recursion practice problems with solutions
Quick quiz
  • Q1. What happens if a recursive function has no base case?
  • Q2. Why is Fibonacci’s naive recursive implementation slow?
  • Q3. Name one reason to prefer an iterative solution over recursion.
Show quiz answers
  1. It calls itself forever (or until stack overflow) → infinite recursion.
  2. It recomputes the same subproblems again and again → exponential time.
  3. Iterative versions are often faster, use less memory, and avoid stack overflow.
Week 3 – Stacks, Queues, Hash Maps
Learn how these core data structures work and where they show up in real problems.
>

You’ll see stacks for “undo” and parentheses checking, queues for scheduling and BFS, and hash maps for fast lookups. AI will help you reason about trade-offs and edge cases.

Goals:
  • Understand LIFO (stack) and FIFO (queue) behavior.
  • Know how hashing enables near O(1) lookup in maps/dicts.
  • Use these structures to solve small interview-style problems.
Practice:
  • Implement a stack and queue using arrays or lists in your main language.
  • Solve a “Valid Parentheses” style problem using a stack.
  • Use a hash map/dictionary to count frequencies in an array or string.

Quick quiz
  • Q1. Which structure would you use to implement “undo” in an editor?
  • Q2. Average-case time complexity for hash map lookup?
  • Q3. Why are queues useful in breadth-first search (BFS)?
Show quiz answers
  1. A stack (LIFO).
  2. O(1) on average.
  3. BFS explores nodes level-by-level; a queue naturally stores “next to visit” nodes in correct order.

Prompt Library (You + AI Coding)

A few reusable prompts you can paste into ChatGPT/Claude/Copilot Chat. Customize them over time.

  • Debugging prompt:
    “You are a senior engineer. I’ll paste an error message and the relevant code. Explain what is likely wrong and give 2–3 concrete fixes. Then show a corrected version of the code. Finally, list tests I should run to be sure it’s fixed.”
  • Refactor & document prompt:
    “Please refactor this code for clarity, add docstrings/comments where helpful, and point out any potential bugs or edge cases. Don’t change external behavior.”
  • Test-generation prompt:
    “Here is a function. Generate unit tests for it in <framework>, focusing on normal cases, edge cases, and error conditions. Explain why each test is useful.”
